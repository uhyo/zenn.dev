## 考察

少なくともこのようなコーディングに関して、一定のeffortは必要なのだということが分かる結果となりました。

カテゴリ別スコアが興味深いですね。Effortをlowにしてもほとんどスコアが下がっていないカテゴリもあります。これらはeffortの影響を受けず、モデルの地力が出るカテゴリなのでしょう。

逆に、high→maxでスコアが上がったカテゴリは、lowにすると逆にしっかり下がるということも見て取れます。

コンポーネント設計が下がるのは予想できましたが、Fable 5の強みとされていたアクセシビリティのスコアまでしっかり下がるのが意外な結果でした。

この結果がFable 5の使い方の参考になれば幸いです。

# React Profession Bench 第13回レポート: Fable 5 effort=low — 「安いFable」は偽りの経済

**実施日**: 2026年7月17日
**評価対象**: Claude **Fable 5 effort=low**（新規、n=3）
**比較対象**: Fable 5 **default**（第8回、n=3）／Fable 5 **max**（第10回、n=3）／既存の全モデル順位表
**評価者モデル**: Claude Sonnet 4.6（`sonnet-4.6` にピン留め。過去全レポートと同一）
**スペック数**: 13
**サンプル数**: 3条件すべて **3回**（n=3、計117スペック分）。第8・10回と合わせ、**Fable 5 の low / default / max 完全な effort 三点**が揃った。

## 概要

Fable 5 は本ベンチの頂点（max で 86.67、第10回）だが、**Opus の約2倍の使用量を食う最も高価なモデル**でもある。ゆえに現実的な問いが立つ——**トークンを節約したい人は effort=low に落とすだろう。そのとき Fable 5 はまだ「使える」のか。安く回した Fable は元を取れるのか。**

本稿がその答えである。結論を先に述べる。**元は取れない。effort=low は Fable にとって偽りの経済である。**

- **Fable 5 low は 74.62**。default（81.33）から **−6.72**、max（86.67）から **−12.05** 落ちる。**low に落とした瞬間、Fable の世代的優位も effort の上積みも、まとめて消し飛ぶ。**
- 落ち方が問題の核心だ。**low が削るのは、よりによって Fable の武器そのもの——コンポーネント設計（3.77→3.13）とアクセシビリティ（3.74→3.13）**。一方で状態設計・Effect・型といった土台はほとんど無傷（4.6/4.2/4.5 を維持）。**low の Fable は「構造は正しいが、設計と仕上げの磨きを失った Fable」**である。
- 順位に置くと **Fable low（74.62）は Opus 4.7（75.7）帯**——2世代前の Claude 相当に落ちる。しかも**先週測った GPT-5.6 の全 max 条件（76.7〜80.2）や Sol high（76.95）を下回る**。**Fable の割高な料金を払いながら、より安いモデルを高 effort で回した方が良いコードが出る**、という逆説的な立ち位置になる。
- 信頼性も一段落ちる。low では 39 run 中 **1本が非コンパイル**（008 フォームアクション）。default/max では観測されなかった綻びである。

## Fable 5 の effort 三点曲線

| effort | 総合スコア（n=3） | 前段からのΔ |
|---|---:|---:|
| **low** | **74.62** | — |
| default | 81.33 | **+6.72** |
| **max** | **86.67** | **+5.33** |

Fable の effort 曲線は **1段あたり約 +6 点でほぼ直線的**である。low→max の全幅は **+12.05**——本ベンチで単一モデルの effort だけを動かして観測した中で最大級の振れ幅だ。第10回で「Fable は effort の恩恵を最も多く受ける（Opus より大きい）」と述べたが、その裏返しとして、**effort を削ったときの失点も Fable が最も大きい**。高価な Fable ほど、effort をケチると損失が大きい。

## カテゴリ別（low / default / max, n=3）— low は何を削るか

| カテゴリ | low | default | max | low−default |
|---|---:|---:|---:|---:|
| 状態設計 | 4.64 | 4.69 | 4.82 | −0.05 |
| Effect衛生 | 4.21 | 4.28 | 4.36 | −0.07 |
| **コンポーネント設計** | **3.13** | 3.77 | 4.23 | **−0.64** |
| TypeScript品質 | 4.51 | 4.79 | 4.85 | −0.28 |
| パフォーマンス意識 | 3.44 | 3.79 | 4.03 | −0.35 |
| **アクセシビリティ** | **3.13** | 3.74 | 4.26 | **−0.61** |

この表が「low の Fable とは何か」を定義する。

- **土台は無傷。** 状態設計（−0.05）・Effect衛生（−0.07）は low でもほとんど落ちない。**Fable は low effort でも、状態の置き場所と副作用の設計という根幹は間違えない。** 「動く React」を書く力は effort に依存しない、Fable の地力である。
- **削られるのは仕上げの2カテゴリ。** low が集中的に落とすのは **コンポーネント設計（−0.64）とアクセシビリティ（−0.61）**——奇しくも**第10回で Fable が max effort で頂点を取った、まさにその2カテゴリ**である。第10回の結論は「effort で弱点（設計）を消し、世代の強み（a11y）で勝つ」だったが、**low effort はその両方を同時に手放す**。とりわけ a11y は default 3.74 → low 3.13 と大きく崩れ、**Fable の世代由来の武器だったはずの a11y が、low では GPT-5.6 三層（3.79–4.36）にすら劣る水準まで落ちる**。
- 結論として、**low の Fable は「骨格は Fable、仕上げは二世代前」というキメラ**である。設計の粒度と a11y の作り込みという、Fable に金を払う理由そのものが effort=low で蒸発する。

つまり、**「Fable を安く使いたい」という動機で effort=low を選ぶと、Fable の割高な料金を払いながら、より安価なモデル（GPT-5.6 各層や Opus 4.8 default）を通常 effort で回すより悪いコードを受け取る**ことになる。コスト効率の観点で、これは合理的な選択肢ではない。

## この実験が答えたこと

問い——**「Fable 5 は effort=low でも使えるか」**。答えは明確である。

- **品質の絶対水準としては「一応使える」**：74.62 は破綻した数字ではなく、状態・副作用・型の土台はしっかりしている。プロトタイプや使い捨てには足りる。
- **しかし「Fable を安く使う」という費用対効果としては使えない**：low は Fable の差別化要素（設計・a11y）だけを狙い撃ちで削り、残るのは高い料金で買う二世代前品質である。**節約したいなら、Fable を弱らせるのではなく、最初から安いモデルを選び、それを高い effort で回すべき**だ——その方が同じ、あるいは少ない支出で良いコードが出る。
- **Fable が金額に見合うのは effort を出し切ったときだけ**：第8・10・13回を通じ、Fable の価値は effort に強く連動することが確定した（low 74.6 / default 81.3 / max 86.7）。**Fable は「効かせて初めて元が取れる」モデル**であり、effort=low で回すのは、最も高いエンジンを積んだ車をアイドリングで走らせるに等しい。

## 今後の課題

1. **他の高価モデルの effort=low 曲線**: 「効かせないと損」が Fable 固有か、上位機一般の性質かを見るため、Opus 4.8 の low も測る価値がある。effort 曲線の傾き（1段 +6）がモデル間でどう違うかは、コスト最適化の実務的指針になる。
2. **コスト計測（継続課題）**: 第8回から積み残し。effort×トークンの実測があって初めて、本稿の「偽りの経済」を金額で裏づけられる。low の実支出が default の何割かを測れば、「安いFable」の割の悪さを定量化できる。
3. **low の非コンパイル率**: 008 で1本出た非コンパイルが low 特有の信頼性低下か偶発かは n=3 では断定できない。effort と「一発で通るか」の関係は、実務では品質スコアと同じくらい重要である。
4. **他モデルの n=3 化（継続課題）**: 順位表の Opus 4.6・GPT-5.4・Sonnet 4.6・Haiku 4.5 は依然 n=1。
