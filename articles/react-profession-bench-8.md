また、`<dialog>`を使ったのも初めてだそうです。HTMLのネイティブな機能の活用に一歩踏み出したようですね。

他にも`useSyncExternalStore`の使用箇所増加など着実な進歩が見られますが、一方で`useTransition`の活用には踏み込んでいないなど、まだ改善の余地が多く残っています。

## 考察

前回（Opus 4.8 effort=highとmaxの比較）において、最も大きな差が出たのはコンポーネント設計のカテゴリでした。このことから、effortを上げることはコンポーネント設計力の向上に有効であると結論付けました。

今回、Fable 5になって全体的なスコアの向上が見られるなか、コンポーネント設計能力はあまり伸びていない（場合によっては低下している）ことで、前回の結論が補強されています。

コンポーネント設計はeffort（思考量）に依存する部分が多い能力であることが分かりました。一方で、モデル自体の「地力」によって向上する能力も確かに存在しています。

「Planモードだけ上位のモデルを使用する」というテクニックがありますが、コンポーネント設計もPlan時に行われるであろうことを考えると、Plan時だけeffortを上げることももしかしたら効果的かもしれません。

## 概要

2026年6月にリリースされた新世代フラッグシップ **Claude Fable 5** を、既存13スペックで n=3 評価した。問いは二つある——**(1) Fable 5 は現行最強の Opus 4.8 を超えるのか。(2) 「世代を上げる」ことは「effort を上げる」ことに勝てるのか。**

第7回で、Opus 4.8 は effort=max にすると default から +4.64 伸び、84.03 でベンチ単独トップに立った。今回の Fable 5 は **default effort** での測定である（max 相当の高 effort 版は本稿では測っていない）。したがって本来のフェアな比較軸は「Fable 5 default 対 Opus 4.8 default」だが、ベンチ全体での位置づけを見るため Opus 4.8 max とも並べる。

結論を先に述べる。**Fable 5 は Opus 4.8 default を確かに上回るが、その幅は小さく（+1.95）、Opus 4.8 max には届かない（−2.70）。** 新世代は1ランク前進したが、その前進幅は「同じ世代で effort を max に上げる」効果より小さい。

- 13スペック平均は **Opus 4.8 default 79.38 → Fable 5 81.33（Δ +1.95）**。
- 一方 **Opus 4.8 max（84.03）には 2.70 及ばない**。Fable 5 は「default 超え・max 未満」の中間に着地した。
- 全モデル順位では **Opus 4.8 default < Fable 5 < Opus 4.8 max** の位置に入る。

## Fable 5 が「効く」スペック・「効かない」スペック

default とレンジが重ならない真の差は3本のみ——勝ち2本（010・013）、負け1本（006）。+1.95 という控えめな総合差に見合う、限定的な分布である。

**伸びた:**
- **013 設定画面Undo/Redo（+10.0、レンジ分離）**: 最大の伸び。Fable 5（90–94）が default（80–84）と完全分離し、**max（85.3）すら +6.7 上回る**。本スペックは Fable 5 が唯一 max を明確に超えた一本である。理由は明快で、013 は **「Undo/Redo の状態モデリング＋モダン React イディオム」の試験**であり、これは Fable 5 の最も得意とする領域だからだ。`{past, present, future}` の reducer で全4アクション（reset を単一履歴として記録・新規変更時の redo クリア含む）を正確に処理し、`useSyncExternalStore`（`prefers-color-scheme`）と React 19 の `<Context value>` 短縮記法という**最新イディオムを採用**、5段ネストの preview を prop drilling ゼロで通した。実際このスペックでは状態設計（4.33）・パフォーマンス（4.00）・アクセシビリティ（4.67）がいずれも **max を上回って**いる。
- **010 ツリーファイルエクスプローラ（+7.0、レンジ分離）**: モノリス化しやすい題材で安定して分解できている。ただし max（80.7）には届かない。
- **005（+8.3）・002（+7.7）** も大きく伸びたが、いずれも default のレンジ上端と接しており「完全分離」には一歩届かない。

**伸びなかった・後退した:**
- **009 再利用コンポーネント（−7.3）**: 最大の後退。3サンプルとも 75–76 と低位安定しており、ブレではなく傾向である。状態設計（5.00）・アクセシビリティ（4.00）は default と同等だが、**コンポーネント設計・TypeScript品質・パフォーマンス意識がそろって −0.67**。重要なのは、**Fable も Opus 4.8 default も「同じ本質的ミス」を犯している**点だ——本スペックの最重要シグナルである非同期アクションの `useTransition`（`action` prop）化を、**両者とも発見できず**、手動 `useState` + `setPending` の tier-2 パターンに留まった。つまり −7.3 の差は「片方だけが持つモダン API」ではなく、**共通の設計ミスの周辺にある二次的な作り込みの質**から生じている。Fable は同じ設計をより雑に実装した: `TaskList` ラッパ不在で App にリスト直書き、不要な `onPendingChange` 脱出ハッチ prop、`useMemo`/`useCallback` 規律の欠如（default は `filteredTasks` を `useMemo` 化）、ゆるい型（`onClick?: () => Promise<unknown>` 対 default の `Omit<ButtonHTMLAttributes, 'onClick'>` 拡張）、さらに **Delete ボタンの pending ラベルが通常ラベルと同一でpending 表示が空洞化する機能的バグ**と、`ConfirmDialog` の不安定 ref が毎レンダーで listener を貼り直す effect バグまで抱えた。009 が「再利用コンポーネントの丁寧な抽出と規律」を最も強く問うスペックである以上、Fable の唯一の弱点がここで最大露出した格好だ。
- **006 通知フィード（−2.7、レンジ分離）**: 小幅だが明確な後退。状態設計が落ちた（4.67 → 4.00）一方アクセシビリティは上がっており（3.00 → 4.00）、得意・不得意の出方が default と微妙に異なる。
- **011 ツールチップ（−10.0 vs max）**: default とはほぼ同等（−1.3）だが、max（89.0）との差が全スペック中最大。`createPortal` ベースの作り込みは、ここでも effort 差がそのまま出ている。

## 新 React API の採用状況（Fable 5 対 Opus 4.8）

スコアとは別に、各モデルが生成したソースを横断 grep し、**新しい React API を「13スペック中いくつで採用したか」**を数えた（再試行・サンプル数の偏りを打ち消すため、スペック単位の採用有無で集計）。結果は小幅だが一貫した差を示す。

| API（採用スペック数 / 13） | Fable 5 | Opus 4.8 def | Opus 4.8 max |
|---|:---:|:---:|:---:|
| `useActionState` | 1（008） | 1（008） | 1（008） |
| `useDeferredValue` | 1（002） | 1（002） | 1（002） |
| `useSyncExternalStore` | **2（005, 013）** | 1（005） | 1（005） |
| `<Context value>` 短縮記法（R19） | **1（013）** | 0（`.Provider`） | 0（`.Provider`） |
| native `<dialog>` | **1（003）** | 0 | 0 |
| `useTransition` | **0** | **0** | **0** |
| `useOptimistic` / `useFormStatus` / `use()` / `useId` | 0 | 0 | 0 |

二つの事実が読み取れる。

- **Opus 4.8 は default と max で新 API フットプリントが完全に一致する**。effort を上げても「使う API の種類」は変わらず、max の改善はあくまで既存 API の使い方の精度（コンポーネント分解）で稼いでいる。新 API の採用は effort ではなく世代で決まる。
- **Fable 5 は最新の React 19 イディオムに一歩多く踏み込む**。`<Context value>` 短縮記法・native `<dialog>`・`useSyncExternalStore` のより広い適用——いずれも Opus 4.8 が（default でも max でも）触れなかったものだ。013 の勝因（`useSyncExternalStore` ＋ Context 短縮）と、009 で両者が `<dialog>` を使わず減点された事実が、この表の差と直接対応している。

ただし**最大の空白は3条件すべてに共通**する。`useTransition`（009 の最重要ミス）・`useOptimistic`・`useFormStatus`・`use()` は**どのモデルも一度も使わなかった**。すなわち Fable 5 が進んでいるのは主に**構文的な最新イディオム**であって、**振る舞いを変える高価値な並行 API（`useTransition`/`useOptimistic`）では Opus と同じく未踏**である。世代を上げても、ここはまだ全モデル共通の死角だ。

（注: n=3・13スペックでの 1スペック差であり、統計的に強い差ではなく「一貫した傾向」として読むべきである。）

## コスト効率の注記

Fable 5 は **Opus の約2倍の使用量を消費する**（本実験中の実測感としても、1サンプルが1回の5時間 usage window に収まらず、s2・s3 で各1回ずつ usage limit に到達した。Opus 4.8 default の n=3 では限界到達はもっと稀だった）。

品質差とコストを重ねると、本ベンチに限れば Fable 5 default の費用対効果は厳しい:

- **対 Opus 4.8 default**: 約2倍のコストで +1.95。
- **対 Opus 4.8 max**: Fable 5 の方が高コストにもかかわらずスコアは −2.70 下。

つまり「同じ予算を積むなら、Fable 5 default に払うより Opus 4.8 を max に上げる方が、この13スペックでは費用対効果が良い」可能性が高い。ただし——**これは default effort の Fable 5 に限った話**である。Fable 5 を高 effort で走らせれば像は変わりうるし（max カテゴリ差がコンポーネント設計に集中している以上、effort で埋まる余地は大きい）、コストの厳密計測（トークン・実行時間）も本稿は行っていない。最終的な費用対効果判断は次の課題に委ねる。

## 運用面: probe-to-wake の初実戦

第7回までに整備した中断耐性機構に、本実験で **probe-to-wake**（usage window の早期復帰検知）を追加し、初めて実戦投入した。

第7回までの usage-limit 待機は、リセット時刻が取得できない（空出力で落ちる）場合に **固定で約5時間5分** 眠る方式だった。だが Anthropic の5時間 window はローリングで、しばしばもっと早く回復する。実際 s1 の初回 limit では、固定待機が真のリセットより **約80分も長く**眠ってしまい、手動で起こす必要があった。

そこで待機ロジックを「固定スリープ」から「**ハードキャップ付きポーリング**」に変えた。算出時刻（プロバイダ提示 or 5h05m フォールバック）は**上限**としてのみ使い、5分ごとに軽量プローブを撃って**最初に成功した瞬間に再開**する。本実験での実績:

- **s2 の limit（07:00 UTC 到達）**: ハードキャップ 12:05 に対し、**10:52 にプローブ成功で早期再開**（実際の window 回復は約3時間52分後）。固定待機なら被っていた約73分の空転を回収。
- **s3 の limit（12:00 UTC 到達）**: 同じく **15:52 に早期再開**（約3時間52分後）、約73分を回収。

2回とも真のローリング window は約3h52m で回復しており、**5h05m の固定フォールバックは実態より1時間以上保守的**だったことが裏づけられた。probe-to-wake により、**リセット時刻を一切パースできなくても、空転は最大でもプローブ間隔（5分）以内**に縮む。第7回で手動介入が要った over-wait は、今回は無人で自動回収された。

## 今後の課題

1. **Fable 5 の高 effort 化**: 本稿は Fable 5 **default** のみ。max との差がコンポーネント設計に集中している以上、Fable 5 を高 effort で走らせれば Opus 4.8 max を超える可能性がある。世代と effort の二軸を Fable 5 でも揃えるのが次の筋。
2. **コスト計測**: 「約2倍」は体感・限界到達頻度ベースにとどまる。トークン消費・実行時間を実測し、費用対効果を定量化すべきである。
3. **全モデル共通の `useTransition` 死角**: 本稿の API 採用調査で判明したとおり、`useTransition`・`useOptimistic`・`useFormStatus` は Fable 5 を含むどのモデルも未採用で、これが 009 の頭打ち要因だった。ベンチ側でこの並行 API 採用を明示的に問うスペックを足し、世代でいつ埋まるかを追う価値がある。
4. **他モデルの n=3 化（継続課題）**: 順位表の Opus 4.6・GPT-5.4・Sonnet 4.6・Haiku 4.5 は依然 n=1。
