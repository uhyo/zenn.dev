## 考察

スコアの内訳を見ると、コンポーネント設計やセマンティクス・アクセシビリティの面で堅調な改善が見られます。

筆者が重要視しているReactの新しめのAPIの採用状況については、Opus 4.7と比較して飛躍的な進化は見られなかったものの、悪化もしていないようです。相変わらずですが、Suspense for data fetchingを自発的に使用することがない一方で`useDeferredValue`は安定して使用できる挙動が不思議ですね。AIもReactを雰囲気で書いているのかもしれません。

今回からn=1ではなくn=3でのベンチマークを行うことにしました。これで数字の信頼性が従来よりは増すでしょう。世間ではOpus 4.7と4.8との違いが分からない、劣化した、といった声も少し聞こえますが、React力をこのように評価すると、しっかりと数字に差が出るのが面白いですね。

## 最重要の発見: モダンAPI採用の「後退」

前回レポート（第4回）で Opus 4.7 の最大の進歩として強調したのは、Claude Opus として**初めて** `useSyncExternalStore`（スペック005）と `useActionState`（スペック008）を適切な文脈で自発採用した点だった。**Opus 4.8 はこの両方を手放した。**

### スペック005 システムステータス監視（−10、72→62）

今回最大の回帰。実装を調べたところ、**`useSyncExternalStore` の使用箇所はゼロ**であり、代わりに `useEffect` が **8箇所**使われていた。4.7 default では同スペックで `useSyncExternalStore` を5箇所使い、オンライン状態・ビューポート・メディアクエリ・visibilitychange・イベントバスの購読を正しくこのフックで実装していたのと対照的である。

評価者も Effect衛生に 2/5 を付け、「このスペックの主眼は `useSyncExternalStore` だが、5つの外部購読すべてが（旧来の effect ベースで）実装されている」と明記した。状態設計・コンポーネント設計・アクセシビリティは 4/5 と悪くないため、回帰のほぼ全量がこの API 選択の後退で説明される。

### スペック008 フォームアクション（−6、76→70）

同様に、**`useActionState` の使用箇所はゼロ**。4.7 default では React 19 の `<form action={formAction}>` + `useActionState` パターンを採用し Effect衛生 5/5・スコア76を記録したが、4.8 は6つの `useState` を手動で束ねる素朴なフォーム実装に戻り、コンポーネント設計は 2/5（単一の約170行 `App`）に低下した。なお TypeScript コンパイルは両世代とも失敗（compiles=false）のまま採点されている。

## 考察

### Opus 4.8 の性格

Opus 4.8 は、Opus 4.7 に対して**「平均的な底力はわずかに向上したが、先進機能の自発採用はむしろ後退した」**という、やや意外なプロファイルを示した。

1. **地力の安定**: 状態設計・コンポーネント設計・アクセシビリティが揃って小幅改善し、モノリス化スペック（006, 010）での分解品質が改善した。一発で全13スペック成功した安定性も含め、「平均的な実装の質」は確かに上がっている。
2. **モダンAPIからの撤退**: 一方で 4.7 の象徴だった `useSyncExternalStore` / `useActionState` を両方手放した。これは訓練データの構成変化、あるいは「より保守的で確実に動くコードを書く」方向へのチューニングを示唆する可能性がある。実際、008 のコンパイル失敗は両世代共通だが、005 では「動く effect 実装」を選んだことでスコアを落とした——*動作の確実性と評価軸（モダンAPIの適切な採用）が衝突した*典型例といえる。

### 解釈上の注意

平均差 +0.93 は、過去の同一モデル再走で観測された run-to-run ノイズ（±10/スペック、平均で ±1〜2）と同程度の大きさである。したがって「4.8 は 4.7 より総合的に優れている」と断言するには、本来は複数回再走による標準偏差の確認が望ましい。ただし、005・008 のモダンAPI後退は**実装を直接 grep して確認した確定的事実**であり、ノイズではない。

### 今後の課題

1. **Opus 4.8 × effort=max**: 4.7 max（78.2）との比較は、予算が許せば本ラウンドで最も価値のある追試である。effort を上げれば 005 の `useSyncExternalStore` が復活するかは特に興味深い。
2. **モダンAPI後退の追跡**: `useSyncExternalStore` / `useActionState` の不採用が 4.8 の一貫した傾向なのか、単一サンプルのブレなのかを、005・008 の複数回再走で切り分けたい。
3. **evaluator JSON パース対策**: 今回は偶然発生しなかったが、根本対策（justification の base64 化等）は依然未実装であり、前回まで5回以上発生している既知のリスクとして残っている。

## 解釈

### Opus 4.8 の性格（改訂後）

Opus 4.8 は、Opus 4.7 に対して **6カテゴリすべてで小〜中幅に向上した、素直な世代交代**である。とりわけコンポーネント分解と TypeScript の堅牢性が伸び、モノリス化しがちだったスペック（006・010）で安定して構造的なコードを書くようになった。モダンAPIの採用傾向は 4.7 と同等で、後退は確認されなかった。

### 方法論上の教訓: n=1 の危うさ

本ラウンド最大の学びは、**ベンチマーク結論そのものではなく、それを導く過程にある**。

- 4.8 初回 n=1 の総合平均（76.31）は、3回中もっとも低い外れ値だった。これだけを見て「4.8 はほぼ横ばい」と結論づけたのは誤りだった。
- 005 はサンプル間で 62〜77 の **15点**、008 は 70〜83 の **13点**、002 は 74〜85 の **11点** も振れる。旧レポート自身が「±10/スペックの run-to-run ノイズ」を警告していたにもかかわらず、その警告の射程を過小評価していた。
- 「grep で確認した確定的事実」も、サンプルが1つなら一般化できない。s1 で `useSyncExternalStore` が0だったのは事実だが、「4.8 が手放した」という命題は偽だった。
- **n=1 が当たるか外れるかは事前に分からない。** 同じ条件でも、4.7 の初回（75.38）は n=3 平均（75.72）にほぼ一致する「当たり」だった一方、4.8 の初回（76.31 vs 79.38）は大きく外した。比較する両モデルの片方だけが外れ値を引けば、差の符号すら逆転しうる（実際、旧レポートの 010 は +7.7→+1.0、全体は +0.93→+3.67 と動いた）。だからこそ**両モデルを揃って n=3 にする**ことが要点であり、評価対象だけ n=3 にしても不十分である。

スコアの揺れが大きいこのベンチマークでは、**世代間比較には両モデルとも最低 n=3 が必要**であることが、身をもって確認された。

## 今後の課題

1. **Opus 4.7 の n=3 化（完了）**: 比較対象の 4.7 も2回追試し、両モデルとも n=3 に揃えた。結果、4.7 の n=3 平均は 75.72（初回 75.38 とほぼ同値）に確定し、世代間差は +4.00 から **+3.67** に微修正された。4.7 の初回サンプルは偶然ながら代表的だった。
2. **他モデルの n=3 化**: 主役の 4.7／4.8 は揃えたが、Opus 4.6・GPT-5.4・Sonnet 4.6・Haiku 4.5 などの順位表の数値は依然 n=1 のままである。順位の僅差部分（GPT-5.4 71.5 と Opus 4.6 70.2 など）は n=3 化で入れ替わる余地がある。
3. **Opus 4.8 × effort=max**: 予算が許せば。4.8 default が既に 4.7 max を上回ったため、4.8 max は新たな全体トップ候補。
4. **レート上限への対処**: 今回の追試では複数回 usage limit（`exited with code 1`／空stderr）に達し、失敗スペックを小バッチで再試行して補完する運用を強いられた。13スペック×複数回を一気に走らせると上限に達しやすいため、**スペック単位の自動リトライ・バックオフ機構**があると運用が安定する。
5. **evaluator JSON パース対策**: 本ラウンドで根本対策を実装済み（`repairJsonEscapes`: 文字列内の不正なエスケープ・制御文字を許容的に修復するフォールバック）。今回は発火しなかったが、過去5回以上発生していた既知のリスクに対する保険として常設した。
