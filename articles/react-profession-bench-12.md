## 考察

Effort=max同士の比較という、前回より客観的に見ても公平な条件で比較しましたが、むしろmax同士の比較ではGPT-5.6とClaude Fable 5 / Opus 4.8との差がさらに広がりました。

GPT-5.6では、**effortによるスコアの伸びがClaudeよりも小さい**ことが明らかになりましたね。Claudeは、Opus 4.8では+4.6、Fable 5では+5.3の伸びがありました。

特に、「effortを上げるとコンポーネント設計が大きく改善する」というのがClaudeでのベンチマークにより確立された法則でしたが、GPT-5.6にはその法則が当てはまらないようです。コンポーネント設計も含めて全体的に伸びてはいますが、他のカテゴリと同程度の伸びであり、コンポーネント設計に特別“効く”ことはありませんでした。

むしろ、元々強みだったアクセシビリティが、effort向上でまだ伸びる余地があったことに驚きです。

Effect衛生のスコアがeffortでまったく伸びず、Lunaに至ってはなぜか下がっていることも注目に値します。GPT-5.6は、全体的にReactの書き癖がモデル自体に染みついていて思考による改善の余地が小さく、それがClaudeよりも少し下手なように見えますね。

第1回の記事でも述べたとおり、このReact習熟度ベンチマークは、筆者がReactにおいて重要視していることをベンチマークに落とし込んだものです。つまり、筆者好みのReactコードが書けるかということです。

この勝負では、勝者は明確なようですね。

**実施日**: 2026年7月16日

**比較対象**: 同三層の **effort=high**（第11回、n=3）／Claude **Opus 4.8 max**・**Fable 5 max**（第7・10回、n=3）／既存の全モデル順位表

## 概要

第11回で GPT-5.6 三層を effort=high で測り、**Sol 76.95 ≈ Luna 76.49 > Terra 73.44** という非単調な序列と、**コンポーネント設計（2.90–3.33）に集約する Claude 上位との構造差**を確認した。そして問いを残した——**max effort でこの差は埋まるのか。第10回で Fable 5 がまさに max effort でコンポーネント設計を 3.77→4.23 へ引き上げて頂点を取ったように、GPT-5.6 も同じ道を辿るのか。**

本稿がその答えである。結論を先に述べる。**埋まらなかった。**

- **effort=max で三層とも上がったが、上げ幅は Sol +3.23 / Terra +3.26 / Luna +1.67（平均 +2.7）**にとどまり、**Claude の effort 効果（Opus +4.6・Fable +5.3、第10回）より小さい**。
- 最上位 **Sol max は 80.18 で GPT-5.6 の最高値**だが、それでも **Fable 5 の *default*（81.3）にすら届かず**、Claude の max フロンティア（Opus 4.8 max 84.0・Fable 5 max 86.7）には **−4〜−6.5** 及ばない。
- 決定的なのは、**コンポーネント設計が max でもほとんど動かなかった点**（三層で +0.18〜+0.28、到達値 3.08–3.62）。Fable が max で 4.23 まで伸ばしてギャップを消したのとは対照的に、**GPT-5.6 の構造的弱点は effort では買えなかった**。
- 生成コードを解析すると理由が見える。**GPT-5.6 は React 19 の並行・アクション系 API（`useTransition`・`useActionState`・`useOptimistic`）をほぼ一切使わず、代わりに `useMemo`/`useCallback` による手動メモ化を多用する「React 18 世代の書き癖」**を持つ。フォームアクション専用スペック（008）ですら `useActionState` の使用は **0 本**だった。

- **肝心のコンポーネント設計は動かない。** max effort での伸びは +0.18〜+0.28 にとどまり、到達値は **3.08–3.62**。Claude 上位の **4.21–4.23** には依然 **0.6〜1.1 の大差**が残る。第10回で Fable が max effort でここを 3.77→**4.23** へ跳ね上げ、Opus max と横並びにして頂点を取ったのとは**正反対**である。**GPT-5.6 にとってコンポーネント設計は「思考量で買える差」ではなく、世代に染みついた書き癖の差**であることを示唆する。
- **max の思考量はアクセシビリティと型に回った。** 三層で最も伸びたのは a11y（Terra +0.54・Luna +0.26・Sol +0.18）と TypeScript（Terra +0.41）。とりわけ Terra は a11y・型のキャッチアップで +3.26 を稼いだ。**GPT-5.6 の max は「弱点（設計）を埋める」のではなく「既に得意な領域（a11y・型）をさらに磨く」方向に流れる**。これが Claude より effort 効果が小さい理由である——伸びが総合を動かしにくい場所に集中している。

## モダン React API の使用実態 — なぜ設計が伸びないか、コードが語る

スコアの背後を、生成コードそのもので裏づける。GPT-5.6 の全 run（各条件の成果物ディレクトリ）を走査し、React 19 世代の API の採用状況を数えた。

**並行・アクション系 API（React 19 の要）はほぼ不在:**

| API | GPT-5.6 での出現 | 備考 |
|---|---|---|
| `useTransition` | **0 / 178 run** | 全条件で一度も使わない |
| `useActionState` | **0**（008 含む） | フォームアクション専用スペックでも 0 |
| `useOptimistic` | 1 | ほぼ皆無 |
| `use()` フック | 0 | 不使用 |
| `useEffectEvent` | 0 | 不使用（実験的なので妥当） |
| `startTransition` | 4 | 稀 |
| `useDeferredValue` | 14 | フィルタ系スペックで散発 |
| `Suspense` | 13 | 散発 |

**代わりに使うのは React 18 世代の手動メモ化イディオム:**

| 条件 | `useMemo` 採用率 | `useCallback` 採用率 | 並行/アクションAPI採用 |
|---|---:|---:|---:|
| GPT-5.6 Sol max | 64% | 64% | 2 / 39 run |
| GPT-5.6 Terra max | 69% | 58% | 1 / 39 run |
| GPT-5.6 Luna max | 60% | 51% | **0 / 43 run** |
| 参考: Fable 5 max | 50% | 67% | 13 / 89 run |
| 参考: Opus 4.8 max | **19%** | 34% | 4 / 152 run |

二つの事実が読み取れる。

1. **GPT-5.6 は React 19 の並行・アクション系 API を実質的に採用しない。** max 条件での並行/アクションAPI採用は Sol 2/39・Terra 1/39・**Luna 0/43**——率にして約 2%。Fable 5 max の 15% と比べて桁違いに少ない。最も象徴的なのが **008 フォームアクション**で、**Claude（Opus・Fable とも一部サンプルで `useActionState` を採用）に対し、GPT-5.6 は三層 max 全 run で `useActionState` 0**。GPT-5.6 は `useState` と手書きハンドラでフォーム状態を組む、React 18 的な作りに終始する。
2. **その穴を手動メモ化で埋めている。** GPT-5.6 三層は `useMemo` を 60–69% の run で使う。これは React Compiler 前提で「既定ではメモ化しない」方針を採る **Opus 4.8 max の 19% を大きく上回る**（本ベンチの anti-pattern カタログは不要なメモ化を減点対象とする）。**GPT-5.6 は「並行 API で構造的に速くする」のではなく、「古い流儀で手動最適化する」ことを選ぶ。**

この書き癖が、スコアのコンポーネント設計（3.1–3.6）とパフォーマンス意識（3.5–3.8）が伸び切らない機序そのものである。**GPT-5.6 は動作する React を確実に書くが、その語彙が React 18 世代で止まっている。** max effort をもってしても語彙は現代化されず、だからコンポーネント設計は動かなかった。第8回以来「全モデル共通」とされてきた `useTransition` の死角は、**GPT-5.6 においては死角というより世代的な地層**であり、Claude 上位（とりわけ Fable）がそこを部分的に踏み越え始めているのと対照をなす。

## 運用面

Sol は最上位機×max effort で本 GPT 実験中最も重く、複数回の usage window 跨ぎを probe-to-wake で自動補完した。Codex バックエンドは budget-cap 機構を持たない（`--max-budget-usd` は claude 専用）ため、max effort でも予算切れ由来の失敗はゼロ——コストは usage としてのみ効く。実行順を **Sol→Terra→Luna（重い順）**としたのは、OpenAI が販促目的で usage 上限を随時リセットする挙動を踏まえ、最重量を先に流し切るためである。中断耐性機構は新バックエンドでも無改修で機能し続けた。

## 今後の課題

1. **設計の粒度を直接測るスペック**: GPT-5.6 と Claude 上位を分ける唯一にして最大の差はコンポーネント設計に集約する。「再利用単位への分解・責務分離」を単独で問う軽量スペックを足せば、モデル間のこの差をより高解像度で追える。
2. **`useActionState`/`useOptimistic` を要求するスペックの明示化**: 008 は結果的にモデルの API 語彙を測る良い試金石になった。React 19 のアクション系を正面から要求するスペックを整備すれば、「世代的な書き癖」を定量化できる。
3. **コスト計測（継続課題）**: 第8回から積み残し。Codex CLI は claude と課金系が異なり、両者を同一軸で比べる方法自体が課題。GPT-5.6 の effort 効果が小さい（+2.7）以上、「品質／コスト」で見たときの max の割の悪さも定量化したい。
4. **他モデルの n=3 化（継続課題）**: 順位表の Opus 4.6・GPT-5.4・Sonnet 4.6・Haiku 4.5 は依然 n=1。
