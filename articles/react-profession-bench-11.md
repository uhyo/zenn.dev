## 考察

まず、GPT-5.6の傾向はClaudeとは異なっていることが分かりました。ClaudeのモデルはTierが上がると各カテゴリのスコアが万遍なく上がる傾向にありましたが、GPT-5.6ではLunaがなぜかSolと競っており、Terraが明確に劣っています。

ただ、スペックごとに見ると、簡単・軽量なスペックではLunaが優位で、重量級スペックではSolが有利という傾向があるようです。高難度なタスクには大きいモデル、という基本はGPT-5.6でも変化していないことが分かります。Terraは何しに来たのでしょうか。

Claudeとの能力の比較は明確です。GPT-5.6はコンポーネント設計でClaude勢に劣る一方で、アクセシビリティに強みを持っています。GPT-5.4の時代から同世代のClaudeモデルよりアクセシビリティが強い傾向を持っていましたが、GPT-5.6にもその強みが受け継がれています。

今回はeffort=highで計測・比較を行いました。普段Codexを使っていないのでGPT-5.6のeffort=highがClaudeのeffort=highと同等なのかはよく分からないので、もしかしたらGPT-5.6側に不利な設定なのかもしれません。

次回はeffort=maxで計測を行い、GPT-5.6の能力を見極めたいと思います。特に、Claudeではコンポーネント設計能力がeffortの影響を大きく受けることが分かっていますから、effort=maxでGPT-5.6が挽回するかもしれません。

次回もお楽しみに。

## 概要

本ベンチはこれまで Claude 一族を中心に測ってきたが、今回初めて **OpenAI の最新機 GPT-5.6 を、その純正 Codex CLI 経由で本格計測**した。GPT-5.6 は Claude の Opus/Sonnet/Haiku に相当する **Sol（最上位）/ Terra（中位）/ Luna（最小）** の三層構成で、いずれも effort を持つ。本稿はまず全三層を **effort=high** で n=3 計測した結果である（effort=max は次稿）。

結論を先に述べる。**予想外の序列が出た。**

- **Sol 76.95 ≈ Luna 76.49 > Terra 73.44**。**最上位の Sol と最小の Luna がほぼ同点で、中位の Terra が最下位**という、**モデルサイズに対して非単調な序列**である。
- 三層とも n=3 のブレは極めて小さく（各サンプル平均のレンジは Sol 76.2–78.1、Terra 73.2–73.6、Luna 76.2–77.0）、この序列は**ノイズではない**。
- GPT-5.6 に共通する弱点は明確に**コンポーネント設計**（三層で 2.90–3.33）で、Claude 勢（Fable/Opus の 3.77–4.23）に対する構造的な差はここに集約する。Terra が最下位なのは、この弱点が三層中で最も深い（2.90）ためである。

- **Terra が最下位である理由＝コンポーネント設計の底（2.90）。** Terra は状態・Effect・パフォーマンスでは Luna/Sol と横並びなのに、**コンポーネント設計が三層中で最も低く（Luna −0.43、Sol −0.20）**、加えて型（4.33）・a11y（3.79）でも僅かに劣る。この一点集中の弱さが総合 −3 の谷を作っている。

- **effort 水準の注記（重要）。** 本稿は GPT-5.6 を **effort=high** で測っている。順位表の Claude 系「default」列は各 CLI の既定 effort での値であり、「max」列（Opus 4.8 max 84.0・Fable 5 max 86.7）はさらに上の effort 帯である。したがって**フェアな最終比較は「GPT-5.6 max 対 Claude max」**であり、それは次稿に譲る。本稿が確定させたのは「**high effort 帯での GPT-5.6 の地力**」——それは Opus 4.7〜Sonnet 5 級、という一点である。

## Claude 勢との差はどこにあるか

GPT-5.6 三層のカテゴリ平均を Claude 上位（第10回のカテゴリ表）と並べると、差の在り処が一点に絞れる。

- **状態設計・Effect・型**: GPT-5.6（4.3–4.7）は Claude 上位（4.3–4.85）と実用上ほぼ横並び。ここでは差がつかない。
- **コンポーネント設計**: GPT-5.6 **2.90–3.33** に対し、Fable 5 default **3.77** / Opus 4.8 max **4.21** / Fable 5 max **4.23**。**約 0.5–0.9 の明確な差**で、これが総合差の主因である。GPT-5.6 は動く React は書くが、**再利用単位への分解・責務の切り分けという「設計の粒度」で Claude 上位に一段譲る**。
- **アクセシビリティ**: Sol（4.36）は Fable 5 max（4.26）すら上回る一方、Terra（3.79）・Luna（3.90）は中位。a11y は層による振れが最も大きいカテゴリで、Sol の数少ない明確な武器である。

要するに、**GPT-5.6 と Claude 上位フロンティアの差は「コンポーネント設計」にほぼ集中する**。第10回で Fable 5 が max effort によってまさにこのコンポーネント設計を 3.77→4.23 へ引き上げてトップを取ったことを思えば、**GPT-5.6 が max effort でこの一点をどこまで埋められるか**が、次稿の最大の見どころになる。

## 運用面: Codex CLI バックエンドの新規実装

本実験のために、ランナーへ **`codex` バックエンドを新設**した（Claude / Copilot に次ぐ三つ目）。要点のみ記す。

- **実装経路**: `codex exec`（非対話）。プロンプトを stdin で渡し、`--skip-git-repo-check`（サンドボックス作業ディレクトリは tmpdir 上の複製で git 管理外）＋ `--dangerously-bypass-approvals-and-sandbox`（ランナーが既にリポジトリ外へ隔離済みのため、Claude の `bypassPermissions` / Copilot の `--allow-all` と同じ姿勢）＋ `--ephemeral`。**effort は `-c model_reasoning_effort="<effort>"`** で指定する。
- **レジストリ**: `gpt-5.6-{luna,terra,sol}-{high,max}` の6キーをバージョンピンで追加。最終目標（三層 × high/max）を過不足なく張った。
- **usage-limit プローブのバックエンド対応**: 従来 `cheapProbe` は `claude` 決め打ちで、codex の usage limit を**別アカウント（claude）に問い合わせて誤って健全判定**しかねなかった。同一バックエンドで探るよう `ModelSpec` を経路全体に通した。codex は成功で exit 0、レート制限を含むエラーで `ERROR: {..."status":429...}` を吐いて exit 1 するため、既存の 429 マーカーがそのまま効く。
- **検証**: 本走前に単一スペック（001）で codex 実装 → コンパイル → Sonnet 4.6 評価 → スコア確定の全経路を通し、パイプラインの健全性を確認してから n=3 を投入した。

Luna は effort=high でも usage limit にほぼ当たらず快調だったが、最上位 Sol は当然ながら重く、複数回の window 跨ぎを probe-to-wake で自動補完した。中断耐性機構（第7〜10回で整備）は新バックエンドでも無改修で機能した。

## 今後の課題

1. **GPT-5.6 effort=max（最優先）**: 本ベンチの最終目標は三層 × max の計測。**フェアな effort 揃えの比較（GPT-5.6 max 対 Opus 4.8 max / Fable 5 max）**はそこで初めて成立する。焦点は「max effort が GPT-5.6 のコンポーネント設計（2.90–3.33）をどこまで押し上げるか」——第10回で Fable がまさにここを埋めて頂点を取った再現・反証になる。
2. **非単調序列の追試**: 「中位 Terra が谷」という結果が max effort でも保たれるか。effort を上げると層の序列が単調化する可能性がある。
3. **コスト計測（継続課題）**: 第8回から積み残し。GPT-5.6 三層 × effort のトークン・実行時間を実測し、Claude 勢と「品質／コスト」で並べたい。Codex CLI は claude と課金系が異なるため、両者を同一軸で比べる方法自体が課題である。
4. **他モデルの n=3 化（継続課題）**: 順位表の Opus 4.6・GPT-5.4・Sonnet 4.6・Haiku 4.5 は依然 n=1。
