## 概要

第8回で Fable 5 の default を測り、「新世代 Fable 5 は Opus 4.8 default を +1.95 上回るが、max effort の Opus 4.8（84.03）には −2.70 届かない。その差はほぼ全部コンポーネント設計に集約し、これは effort で買える種類の差だ」と結論した。そして最後に一つ問いを残した——**では Fable 5 自身を max effort で走らせたら、その差は埋まり、Opus 4.8 max を超えるのか。**

本稿がその答えである。結論を先に述べる。**埋まった。そして超えた。**

- **Fable 5 max は 86.67 で、本ベンチの新たな単独トップ**。Opus 4.8 max（84.03）を **+2.64** 上回る。
- 第8回で Fable の唯一の弱点だった**コンポーネント設計は、max effort で 3.77 → 4.23 へ跳ね上がり、Opus 4.8 max（4.21）と完全に横並びになった**。第8回の仮説（「足りないのは能力ではなく思考量、effort で埋まる」）は的中した。
- 弱点が消えた結果、**Fable の地力であるアクセシビリティ（+0.52）が決定打**として残り、これが Opus 4.8 max を抜いた主因である。

## 「世代 × effort」の 2×2

本ベンチはこれで、Opus 4.8 と Fable 5 の2世代を default／max の2水準で揃えた**完全な 2×2** を持つに至った。総合スコア（n=3平均）は次のとおり。

| | default | **max** | Δ(effort) |
|---|---:|---:|---:|
| **Opus 4.8** | 79.38 | 84.03 | **+4.6** |
| **Fable 5** | 81.33 | **86.67** | **+5.3** |
| **Δ(generation)** | **+1.9** | **+2.6** | |

この表が本稿の全てを語る。

- **effort の効果（縦、+4.6〜+5.3）は世代の効果（横、+1.9〜+2.6）より大きい**。この13スペックでは、**「世代を1つ上げる」より「effort を max に上げる」方が効く**。第7回で Opus について観測した傾向が、Fable でも（むしろより強く）再現した。
- **Fable は effort の恩恵をより多く受ける（+5.3 対 Opus の +4.6）**。第8回で見たとおり default Fable はコンポーネント設計に伸びしろを残していたため、max effort で埋める余地が Opus より大きかった。
- 2要因はほぼ加算的で、わずかに正の交互作用（+0.7）がある。Opus 4.8 default を起点に世代（+1.9）と effort（+4.6）を単純加算すると 85.9、実測は 86.67——**世代と effort は概ね独立に積み上がる**。

## 009 の名誉回復——バイアス除去と effort の二重効果

第8回で Fable の最大の後退とされた **009 再利用コンポーネント**は、本稿で完全に評価が覆った。

- default: **75.3**（Opus 4.8 def 82.7 に −7.3、当時「最大の後退」）
- max: **85.7**（Opus 4.8 max 82.3 を **+3.3 上回る**）

+10.4 の反転である。これは二つの要因が重なった結果だ。

1. **予算バイアスの除去。** 009 は本スイート最多のファイル数を要する最重量スペックで、max effort だと当初の一律 `$5` 予算上限を**実装の途中で使い切って落ちていた**。約16回の試行のうち唯一 $5 に収まった「最も手を抜いた」出力だけが生き残り、s1 のスコアを 73 に押し下げていた（選択バイアス）。予算上限を effort に応じてスケールさせ（max は $15）、009 を単独再実行したところ **73 → 82** に是正された。
2. **effort による本質的改善。** 009 は「再利用コンポーネントの丁寧な抽出＋メモ化規律」を最も強く問う、まさに Fable default の弱点を突くスペックだった。max effort はその弱点を直接埋めた。s3 では 90 に達している。

第8回で指摘した「Fable と Opus は同じ本質的ミス（`useTransition` 不使用）を共有し、差は二次的な作り込みの質から生じる」という構図のうち、**二次的な作り込みの質が max effort で Opus 以上に引き上がった**格好である。

## コスト効率の注記

Fable 5 max は**本ベンチ最高スコアであると同時に、最も高コストな条件**でもある。Fable 5 は Opus の約2倍の使用量を消費し、そこに max effort が重なる。実運用上の負荷は第7回・第8回を大きく超えた:

- 単一スペックが `$5` を超過するため予算上限を `$15`（max 時）へ引き上げる必要があった（上記 009）。
- 全13×3サンプルは1回の usage window にまったく収まらず、**数日にわたり多数の usage limit を跨いだ**。Fable 5 には Opus より厳しい専用の usage 上限があるらしく、リフィルにも通常の5時間 window より長くかかる場面があった。

したがって費用対効果の観点では、**+2.64 のためにどれだけ払えるか**は用途次第である。「絶対品質の最大化」を求めるなら Fable 5 max が現時点の頂点だが、「コスト当たりの品質」では Opus 4.8 max や Fable 5 default が依然として合理的な選択肢である。本稿は「Fable 5 max が最も良いコードを書く」ことは示すが、「割に合うか」はコスト条件に委ねる。

## 運用面: 中断耐性機構の総合試験

本実験は、これまで整備してきた復旧機構すべてが同時に試された総決算だった。

- **予算上限の failure kind 化**: 「`Error: Exceeded USD budget` で code 1 終了」を独立の `budget` 失敗として分類し、無駄な transient 再試行や usage-limit 待機に落ちないよう修正（009 が数パスを空振りしていた問題を解消）。予算上限は effort 連動（default $5／max $15）＋ `--max-budget-usd` で上書き可能とした。
- **probe-to-wake**: reset 時刻が取れない usage limit でも、5分ごとの軽量プローブで**リフィルの瞬間に自動再開**。本実験だけで多数回発火し、固定待機に比べ延べ数時間の空転を回収した。
- **incremental scores + `--resume` + 選択的再実行**: s1 の 009 だけを外科的に再実行（予算バイアス除去）できたのは、スペック単位でスコアを永続化しているおかげである。他の12スペックは同一条件（Sonnet 4.6 評価・$5 で完走）のまま保持し、影響を受けた1本だけを差し替えた。

教訓は第7回から一貫する——**長時間ベンチの結論の質は、スコアのブレ（n=3）だけでなく、実行環境のブレ（中断耐性）と計測条件のブレ（予算上限のような静かなバイアス源）にも左右される**。今回はその三つすべてを制御して初めて、86.67 という数字を信頼して置ける。

## 今後の課題

1. **コスト計測**: 第8回から積み残しの宿題。max/default × 世代のトークン・実行時間を実測し、2×2 を「品質」だけでなく「品質／コスト」でも描くべきである。
2. **Fable 5 の usage 上限特性**: Fable 5 専用の usage 上限とリフィル挙動は、probe-to-wake の想定（5時間ローリング）と異なる場面があった。長時間運用の予測精度のため、Fable の window 挙動を別途プロファイルする価値がある。
3. **`useTransition`/`useOptimistic` の死角**: 第8回で観測した「全モデル共通の高価値並行 API 未踏」は max effort でも解消していない（009 の `useTransition` は Fable max でも不使用のまま二次的作り込みで稼いだ）。この死角を明示的に突くスペックの追加が、次世代を測る解像度を上げる。
4. **他モデルの n=3 化（継続課題）**: 順位表の Opus 4.6・GPT-5.4・Sonnet 4.6・Haiku 4.5 は依然 n=1。
