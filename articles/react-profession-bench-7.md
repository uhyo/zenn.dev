## 考察

スコアの詳細を調べると、スコアを押し上げる主な要因が**コンポーネント設計**にあることが分かります。特に、effort=maxにすることでちゃんとしたコンポーネント設計を安定して出してくれるようになります。

「安定」という点がポイントで、effort=highでは確率的に変な設計が出されることがあったのに対し、effort=maxでは複数回の測定でも安定して高水準のスコアが出るようになっています。

上記のスコア表で、effort=maxにより大きくスコアが伸びたスペックは、複雑な設計を要求するものでした。逆に、単純なものについてはeffort=maxにしてもあまりスコアが伸びない傾向が見られました。

総じて、effortを上げることの効果は公式に述べられているとおりで、複雑な設計が必要な場面で設計をやり切る能力が上がることをこのベンチマークでも確認できました。特に、失敗する確率が減るらしいことは重要です。

裏を返せば、怪しい方向に進みそうになったら人間がすぐに止めることができるのであれば、低いeffortでも成果を出せるかもしれません。最近トレンドとなっているAIのトークン節約において、人間が価値を出せる点なのかもしれません。

# React Profession Bench 第7回レポート: Opus 4.8 effort=max は割に合うか

**実施日**: 2026年6月2日
**評価対象**: Claude Opus 4.8 **effort=max**（新規、n=3）
**比較対象**: Claude Opus 4.8 **effort=default(high)**（第6回、n=3）
**評価者モデル**: Claude Sonnet 4.6（過去全レポートと同一）
**スペック数**: 13
**サンプル数**: 両条件とも **3回**（n=3、計78スペック分）。同一モデル・同一評価者で、**effort 設定だけを変えた対照実験**である。

## 概要

第6回では予算の都合で見送った **effort=max 実験**を、今回 n=3 で実施した。問いは単純である——**Opus 4.8 で effort を max に上げると、コード品質は実際に上がるのか。上がるとしてどこが上がるのか。**

結論を先に述べる。**max effort は確かに効く。ただし全スペック一律にではなく、「構造化が要るスペック」で効く。**

- 13スペック平均は **default 79.38 → max 84.03（Δ +4.64）**。
- max は **13スペック中10本で default を上回り**、下回った3本（001・007・009）はいずれも誤差の範囲（−1.3／−1.3／−0.3、レンジ完全重複）。
- うち **5本はレンジが重ならない**（max の最小値が default の最大値を上回る）真の改善で、世代交代1回分（第6回の 4.7→4.8 が +3.67）に匹敵する差を effort 変更だけで得た。

このカテゴリ表が、**max effort が「どこに」効くか**を最も雄弁に語っている。

- **コンポーネント設計が突出して伸びた（+0.54）**。max effort の追加思考は、主にコンポーネント分解——「どこで切るか」「props をどう通すか」——という設計判断に投下されている。

## effort=max が「効く」スペック

レンジが重ならない真の改善5本。いずれも「素朴に書くと崩れるが、丁寧に設計すれば伸びる」種類のスペックである。

- **005 システムステータス監視（+13.7）**: 最大の伸び。default では 62〜77 と **15点も振れる最難関スペック**だったが、max では 83〜88 に収束した。max effort の本質的な価値が「平均の底上げ」と「分散の縮小」の両方にあることを示す代表例。
- **002 データダッシュボード（+10.3）**: max（91–93）が default（74–85）と完全分離。`useDeferredValue` を軸にした検索体験とアクセシビリティを、毎回安定して作り込んでいる。
- **010 ツリーファイルエクスプローラ（+10.0）**: モノリス化しやすい題材で、max は毎回きちんとコンポーネント分解する。カテゴリ表の「コンポーネント設計 +0.54」がスペック単位で現れた一本。
- **011 ツールチップ/ポップオーバー（+8.7）**: max（87–90）が default（79–81）と分離。`createPortal` ベースの実装とアクセシビリティの作り込みが安定。
- **012 マルチタブエディタ（+6.0）**: **3回とも 98 点**という驚異的な安定度。本ベンチ全78サンプル中の最高スコアであり、max effort が複雑なスペックで上限性能を引き出せることを示す。

## effort=max が「効かない」スペック

max が下回った3本（001・007・009）は、いずれも Δ が誤差（≤1.3）でレンジも完全に重なり、**有意な後退ではない**。むしろ注目すべきは、これらが「効かない」場所の性格である。

- **001 イベント登録フォーム（−1.3）**: 単純なフォーム。default でも 90〜94 と高く、設計判断の余地が少ない。max は1サンプルだけ 85 に沈み、**簡単なスペックでは追加思考がむしろ揺らぎを生む**可能性すら示唆する。
- **007 SNSフィード（−1.3）・009 再利用コンポーネント（−0.3）**: 実質同値。

つまり **max effort は「考える余地のあるスペックで考える余地を使い切る」改善**であり、設計判断が少ない単純なスペックや、能力上限で決まる領域（型付け）には効かない。effort を上げれば一律に上がる、というものではない。

## 全モデル順位への位置づけ

```
Haiku 4.5 (61.4) < Sonnet 4.6 (66.5) < Opus 4.6 (70.2) < GPT-5.4 (71.5)
  < Opus 4.7 (75.7, n=3) < Opus 4.8 default (79.4, n=3) < Opus 4.8 max (84.0, n=3)
```

**Opus 4.8 max（84.03）が本ベンチの新たな単独トップ**である。第6回時点では参考値だった「Opus 4.7 max（78.2、n=1）」を大きく上回り、かつ **n=3 で揃った確かな数値**である点が重要だ。effort=max による +4.64 は、世代交代1回分（4.7→4.8 default の +3.67）を上回る。**「世代を上げる」より「effort を上げる」方が、この13スペックでは効果が大きかった**ことになる。

ただし当然ながら max effort は推論コストが高い。+4.64 という伸びがそのコストに見合うかは用途次第であり、本レポートは「効くか」には答えるが「割に合うか」の最終判断はコスト条件に委ねる。

## 運用面: 中断耐性の検証

本実験は、結果と同じくらい**実行プロセス自体が試金石**になった。effort=max の全13スペック×3サンプルは1回の usage window（5時間）にはまったく収まらず、しかも実行ホストがノートPC（WSL2）で、**実験中に複数回のスリープと、Windows Update による不意の再起動**まで発生した。

第6回までに整備した復旧機構が、これらすべてを取りこぼしなく吸収した:

- **incremental scores + `--resume`**: スペック完了ごとにスコアを書き出すため、ハードキル（再起動）でも完了済みスペックは失われない。再起動後は同じコマンドを再投入するだけで続きから走る。
- **silent-limit 検出（probe）**: usage limit は「空出力のまま終了」だけでなく、**12分走った末に部分出力を残して落ちる**形でも現れた。当初の空出力前提の検出ではこれを transient と誤認し、ドライバが3パス分を無駄に空振りしていた。**「非limit失敗時は必ずアカウントを能動プローブする」**よう修正し、本番ログ上で正しく機能することを確認した（spec 013 で発火、アカウント健全→transient 再試行→成功）。
- **transient self-heal / `--keep-awake`**: スリープで切れた接続は短時間リトライで自己回復。`--keep-awake` はアイドルスリープを抑止する（ただし蓋閉じ・OS再起動は防げないため、最終的な保証は self-heal と `--resume` が担う）。

教訓として、**スコアのブレ（n=3 が要る）とは別に、実行環境のブレ（中断耐性が要る）も、長時間ベンチの結論の質を左右する**。今回の max 実験は、その両方を備えて初めて完走できた。

## 今後の課題

1. **他 effort 水準の n=3 化**: 4.7 max は依然 n=1（78.2）。4.8 で示された「max は構造化スペックで効く」傾向が世代に依らないかを見るには、4.7 max も n=3 化して 4.8 max と対照するのが筋である。
2. **コスト計測**: 本レポートは品質差（+4.64）だけを示し、推論コスト差を測っていない。「割に合うか」を論じるには、max/default のトークン消費・実行時間の実測を併記すべきである。
3. **他モデルの n=3 化（継続課題）**: 順位表の Opus 4.6・GPT-5.4・Sonnet 4.6・Haiku 4.5 は依然 n=1。
4. **中断耐性の更なる自動化**: 現状、OS再起動後の再投入は手動（または監視からの再launch）である。頻繁に電源が落ちるホストで完全無人運用するなら、ブート時自動再開（systemd user service / Windows ログオンタスク）まで踏み込む余地がある。
